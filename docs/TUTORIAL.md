# 🎓 初学者教学文档：从零理解这个 ESP32 项目

> **本文目标**：即使你没有任何嵌入式开发经验，读完这篇文档后，也能明白这个项目的**各个模块在做什么**、**为什么这样设计**、以及**代码是如何流转的**。

---

## 一、把项目想象成一家餐厅

为了让你更容易理解分层架构，我们把整个系统比喻成一家**智能餐厅**：

| 系统层级 | 餐厅比喻 | 职责 |
|---------|---------|------|
| **云侧 (大模型 API)** | **总部大厨** | 远程制定菜谱（高层决策），不直接接触厨房 |
| **边侧 (PC + MQTT Broker)** | **餐厅经理 + 前台** | 接收总部指令，协调各分店，处理顾客订单 |
| **端侧 (ESP32)** | **厨房 + 服务员** | 直接做菜（控制电机）、传菜（读取传感器）、用对讲机和前台沟通（MQTT） |

在这个项目里，**ESP32 就是厨房**。厨房里有：
- **灶台** → SG90 舵机（负责转动）
- **氛围灯** → RGB LED（负责显示当前状态颜色）
- **调味旋钮** → EC11 编码器（厨师手动调节角度）
- **对讲机** → WiFi + MQTT（接收经理的远程指令，汇报当前状态）

## 二、为什么代码要分好几层？

想象一下，如果厨师直接用手抓着电线去控制舵机，会发生什么情况？
- 一旦换了一个舵机型号，整个菜谱都要重写。
- 一旦餐厅经理换了对讲机品牌，厨师又要重新学习怎么接线。
- 代码会变得像一团乱麻，谁也看不懂。

所以，我们引入了**分层设计**。

### 2.1 硬件抽象层 (HAL) — "设备的翻译官"

`main/hal/` 目录下的代码，就是**翻译官**。

比如 `hal_servo.c` 这个翻译官的工作是：
> 你说"把舵机转到 90 度"，他负责把这句话翻译成"在 GPIO4 上输出 1.5ms 宽度的 PWM 脉冲"。

**好处**：
- 上层代码（`main.c`）只需要说"转 90 度"，永远不用关心底层 PWM 怎么配置。
- 如果以后舵机坏了，换了一个接在 GPIO10 上的新型号，**只需要改 `hal_servo.c` 里的几行代码**，上层完全不用动。

### 2.2 网络通信层 (NET) — "对讲机专员"

`main/net/` 目录下的代码，是负责操作**对讲机**的专员。

`net_wifi.c` 负责：
> 开机后自动连上餐厅的 WiFi，如果断线了自动重拨。

`net_mqtt.c` 负责：
> 用对讲机（MQTT 协议）向前台汇报："当前角度 90 度，灯是蓝色"。
> 同时也收听前台指令："经理说把角度调到 120 度！"

**好处**：
- 厨房（HAL）不知道对讲机长什么样。就算以后从 MQTT 换成微信/WebSocket，也只需要换对讲机专员（改 `net/` 目录），厨师（`hal/`）该干嘛干嘛。

### 2.3 应用层 (main.c) — "餐厅主厨"

`main.c` 就是这家厨房的大脑。它不做具体接线，也不操作对讲机，它只做**决策和调度**：

1. 开业前，雇佣翻译官和对讲机专员（初始化 HAL + NET）。
2. 设定规则：角度 0 度时灯是绿色，90 度蓝色，180 度红色（颜色映射策略）。
3. 每天早上，服务员（`mqtt_publish_task`）每隔 0.5 秒向前台汇报一次状态。
4. 主厨自己站在灶台边，每 20 毫秒检查一次："当前目标角度变了吗？变了就更新舵机和灯！"

## 三、FreeRTOS 任务是什么？

ESP32 芯片里面有两个 CPU 核心。但如果没有操作系统，程序只能一行一行顺序执行。如果程序卡在"等待 SPI 传输完成"那一步，舵机就会卡住不动，RGB 灯也会 frozen。

为了解决这个问题，我们使用了 **FreeRTOS**，它是一个**实时操作系统**。在 FreeRTOS 里，程序可以拆成多个**任务 (Task)**，就像餐厅里有多个员工同时干活：

```
┌─────────────────────────────────────────────┐
│           FreeRTOS 任务调度器                │
├─────────────────────────────────────────────┤
│  任务 A: ec11_reader (优先级 20, 最高)      │
│     只管读取 EC11 角度，立刻广播给队列        │
├─────────────────────────────────────────────┤
│  任务 B: servo_task (优先级 15)              │
│     每 10ms 将舵机角度 lerp 1°              │
├─────────────────────────────────────────────┤
│  任务 C: rgb_task (优先级 14)                │
│     接收广播，立即更新 RGB 颜色              │
├─────────────────────────────────────────────┤
│  任务 D: lvgl_task (优先级 5)                │
│     EC11 按键时触发 GC9A01 图片旋转          │
└─────────────────────────────────────────────┘
```

**关键概念**：
- **任务可以并行执行**（实际上是在 CPU 之间快速切换，看起来像同时跑）。
- **中断 (ISR)** 优先级最高。当你旋转编码器时，CPU 会立刻放下手里的活，先去处理旋转事件，处理完再回来继续跑任务。
- **`vTaskDelay(pdMS_TO_TICKS(20))`** 的意思就是"这个任务先睡 20 毫秒，把 CPU 让给别人用"。
- **广播-订阅模式**：`ec11_reader` 读取 EC11 后立即广播到队列，所有消费者任务独立订阅，互不阻塞。

## 四、MQTT 是什么？为什么选它？

### 4.1 比喻：MQTT 是一套"广播对讲机系统"

在餐厅里，厨房、前台、经理室都需要沟通。如果每两个人之间都拉一条电话线，线会多得像蜘蛛网。

MQTT 解决这个问题的办法是引入一个**总台 (Broker)**：
- 所有人只和对讲机总台说话。
- 说话的时候指定一个**频道 (Topic)**，比如 `device/esp32_01/commands`。
- 订阅了这个频道的人，就能收到消息。

### 4.2 发布 (Publish) vs 订阅 (Subscribe)

| 动作 | 比喻 | 本项目实例 |
|------|------|-----------|
| **Publish** | 向某个频道喊话 | ESP32 向 `device/esp32_01/sensors` 发送角度数据 |
| **Subscribe** | 监听某个频道 | ESP32 监听 `device/esp32_01/commands`，等待经理指令 |

### 4.3 为什么不用 HTTP？

你可能会问：我平时上网都用 HTTP，为什么这里不用 HTTP 发数据？

因为 HTTP 是"一问一答"模式：
> 客户端："服务器，你现在有什么指令给我？"  
> 服务器："没有。"  
> （1 秒后）  
> 客户端："服务器，你现在有什么指令给我？"  
> 服务器："没有。"

这太浪费了，而且如果服务器突然有指令，它没法主动推给客户端，只能等客户端来问。

MQTT 是**长连接**模式：
> ESP32 和 Broker 之间保持一条打开的 TCP 管道。一旦有新的指令，Broker 立刻推过来，没有任何延迟。同时 ESP32 也可以随时主动上报数据。

对于需要**实时控制**的机器人系统，MQTT 是最佳选择。

## 五、ESP-IDF 是什么？CMake 又在做什么？

### 5.1 ESP-IDF = 乐鑫官方工具箱

ESP-IDF（Espressif IoT Development Framework）是乐鑫公司为 ESP32 芯片提供的**官方软件开发框架**。你可以把它理解为：
> 一个巨大的工具箱，里面已经帮你准备好了 WiFi 驱动、PWM 生成器、GPIO 中断管理、MQTT 客户端等等。

你的代码不需要从零开始写"怎么让 GPIO4 发出 50Hz 的方波"，只需要调用 ESP-IDF 里的现成函数：
```c
ledc_set_duty(SERVO_LEDC_MODE, SERVO_LEDC_CHANNEL, duty);
```

### 5.2 组件系统 (Component System)

ESP-IDF 把各种功能拆成了一个个**组件 (Component)**，比如：
- `esp_driver_ledc`：LED PWM 控制器组件
- `esp_driver_rmt`：远程控制收发器组件
- `esp_wifi`：WiFi 组件
- `esp-mqtt`：MQTT 客户端组件

每个组件都有自己的 `.c` 文件、头文件和 `CMakeLists.txt`。

在 `main/CMakeLists.txt` 里，我们通过 `REQUIRES` 告诉构建系统：
> "我的 `main` 组件需要用到 `esp_wifi` 和 `esp-mqtt`，请把它们一起编译进来！"

### 5.3 CMake 是"建筑蓝图"

CMake 本身不是编译器，它是一个**构建系统的生成器**。它的工作就像建筑蓝图：
1. 读取你的 `CMakeLists.txt`，了解项目有哪些 `.c` 文件、依赖哪些组件。
2. 生成一份详细的施工图纸（这里是 `build.ninja`）。
3. `ninja` 根据施工图纸调用真正的编译器（GCC），把 `.c` 文件编译成 `.o`，再链接成最终的 `blink.elf` 和 `blink.bin`。

**你不需要理解 CMake 的每一个细节**，只需要知道：
> 修改了 `CMakeLists.txt` 里的 `REQUIRES`，就相当于告诉蓝图"我需要多采购一批钢筋（组件）"。

## 六、代码的生命周期：从旋转编码器到舵机转动

这是最精彩的部分。让我们追踪一次完整的用户体验：

**你用手顺时针旋转了一下 EC11 编码器。**

### Step 1: 硬件触发中断
编码器的 CLK 引脚从 1 变成 0（下降沿）。这个电平变化触发了 ESP32 的 **GPIO 中断**。

### Step 2: 中断处理程序 (ISR)
CPU 立刻暂停当前正在跑的任务，跳入 `hal_ec11.c` 里的 `ec11_isr_handler()`：
1. 读取 DT 引脚，发现它是高电平 → 判断为**顺时针 (CW)**。
2. 进入临界区，把全局变量 `s_angle` 加 2（变成 92 度）。
3. 把事件发送到 `hal_ec11_get_queue()` 队列（**非阻塞**）。

### Step 3: ec11_reader_task 接收并广播
`ec11_reader_task`（优先级 20，最高优先级）从队列中收到事件：
```c
angle_broadcast_t msg = {
    .angle = ev.angle,         // 92
    .servo_target = CFG_EC11_TO_SERVO(92),  // 映射后舵机目标角度
    .r = ..., .g = ..., .b = ...,  // 颜色插值计算
    .button = 0,
};
xQueueSend(s_angle_queue, &msg, 0);  // 广播给所有消费者
```

### Step 4: 消费者任务独立响应
**servo_task**（优先级 15）收到消息，每 10ms 将舵机 lerp 1° 逼近目标：
**rgb_task**（优先级 14）收到消息，立即更新 RGB 颜色（无延迟）：
**lvgl_task**（优先级 5）收到消息，更新方块旋转角度，调用 `lv_refr_now()` 刷新：
### Step 5: HAL 层翻译指令
`hal_servo_set_angle(92)` 内部计算 PWM duty 值，调用 `ledc_set_duty()`，SG90 舵机收到新的脉冲宽度，开始转动到 92 度。

同时 `hal_rgb_set_color()` 内部用 RMT 发送 WS2812 时序，RGB LED 的颜色变成了偏蓝的紫色。

**关键改进**：SPI 传输（GC9A01 显示）在独立低优先级任务中执行，每 20 行调用一次 `vTaskDelay(1)` 让出 CPU，永远不会阻塞舵机和 RGB 的响应。

## 七、远程控制的生命周期

现在假设边侧 PC 上的 Python Gateway 调用了大模型 API，大模型决定把舵机调到 120 度。

### Step 1: 边侧下发指令
Python Gateway 向 MQTT Broker 发布消息：
```json
{"angle":120}
```
Topic 是 `device/esp32_01/commands`。

### Step 2: MQTT 事件处理
Broker 立刻把消息推给 ESP32。`esp-mqtt` 内部触发 `MQTT_EVENT_DATA` 事件，调用 `net_mqtt.c` 里的 `mqtt_event_handler()`。

### Step 3: 解析 JSON
`net_mqtt.c` 用轻量级字符串解析器从 `"angle"` 后面提取出数字 `120`。

### Step 4: 调用指令回调
解析成功后，调用 `main.c` 里注册的 `mqtt_cmd_cb(120)`：
```c
g_target_angle = 120;
```

### Step 5: 主循环同步
和本地控制一样，20ms 后 `app_main` 的主循环发现 `g_target_angle` 变成了 120，于是更新舵机和 RGB 颜色。

**神奇之处在于**：无论是本地旋转编码器还是远程 MQTT 指令，最终都是通过**修改同一个变量 `g_target_angle`** 来实现的。主循环不需要知道角度是怎么来的，它只负责"把当前目标角度同步到硬件上"。

## 八、给初学者的小结

### 你应该记住的 3 个设计原则

1. **HAL 层隔离硬件细节**
   - 上层代码只调 `hal_xxx_set_xxx()`，永远不用关心 GPIO 编号和 PWM 配置。

2. **NET 层隔离网络细节**
   - HAL 层不知道 WiFi 是什么，也不知道 MQTT 是什么。网络协议变了，只改 `net/`。

3. **main.c 只做调度决策**
   - 不要在中断里做复杂计算，不要在 HAL 里写业务逻辑。`main.c` 作为胶水层，把本地输入、远程输入、输出执行串联起来。

### 你应该记住的 3 个技术概念

1. **FreeRTOS 任务 = 并行的员工**
   - 多个任务通过 `vTaskDelay()` 轮流使用 CPU，互不阻塞。

2. **MQTT = 广播对讲机**
   - 发布/订阅模型，适合传感器上报 + 远程指令下发。

3. **中断 (ISR) = 紧急插播**
   - 硬件事件（如编码器旋转）会打断当前任务，处理完后恢复原任务。

## 九、下一步你可以尝试什么？

读完本文档后，如果你有兴趣继续探索，可以从以下几个方向入手（从易到难）：

### 🔧 方向 1：修改业务逻辑（最简单，不改 HAL/NET）
- 打开 `main.c`，找到 `angle_to_color()` 函数，把颜色映射改成你喜欢的配色方案。
- 修改 `hal_ec11_set_step(5)`，让编码器每格转动 5 度（更灵敏）。

### 🔌 方向 2：接入新传感器（练习 HAL 扩展）
- 买一个 HC-SR04 超声波测距模块。
- 在 `main/hal/` 下新建 `hal_hcsr04.c/.h`。
- 在 `main.c` 的 `mqtt_publish_task` 里把距离数据也打包进 JSON 发出去。

### 🌐 方向 3：搭建边侧 Python Gateway（进入 Phase 2）
- 在你的电脑上安装 `paho-mqtt` 和 `mosquitto`。
- 写一个 Python 脚本，订阅 `device/esp32_01/sensors`，打印收到的数据。
- 更进一步：把传感器数据发给 OpenAI API，让大模型返回控制指令，再发回 `device/esp32_01/commands`。
